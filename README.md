# بنك المعلومات المؤسسي — تطبيق الويب (المراحل 1–5)

واجهة Next.js للنظام: تسجيل الدخول، ثنائية اللغة عربي/إنجليزي مع RTL/LTR،
قائمة جانبية ديناميكية حسب الدور، وإدارة الهيكل التنظيمي والمستخدمين.

يعتمد على قاعدة بيانات **المرحلة 0** — نفّذ ملفات Migrations أولاً (حزمة `ckb-phase0-database`).

## ما تتضمنه هذه المرحلة

- **تسجيل الدخول** عبر Supabase Auth مع تجديد الجلسة في Middleware، ومنع دخول
  الحسابات الموقوفة (يُسجَّل خروجها فوراً مع رسالة واضحة)
- **ثنائية اللغة**: عربي (RTL، افتراضي) / إنجليزي (LTR) بتبديل فوري من الرأس،
  بخط IBM Plex Sans Arabic المضمّن محلياً (بدون طلبات خارجية)
- **القائمة الجانبية الديناميكية**: تُبنى من أدوار المستخدم الفعلية في قاعدة
  البيانات — قسم «الإدارة» يظهر لمدير النظام فقط، ولا توجد عناصر لوحدات لم تُنفذ بعد
- **الهيكل التنظيمي**: إدارة القطاعات والوحدات (إدارة/قسم/قسم فرعي) بعرض هرمي،
  مع تعيين المدير لكل وحدة — كل ذلك ديناميكي من الواجهة
- **المستخدمون**: إنشاء الحسابات (Auth + Profile في خطوة واحدة عبر Service Role
  على الخادم فقط)، إسناد الأدوار الأربعة (مع تحديد قسم مدير القسم)، إيقاف الحساب
  يُنهي جلساته فوراً، وكل عملية تُسجل في سجل التدقيق
- **الهوية البصرية**: هوية الماجدية المعتمدة — الفحمي `#2A3335` للشريط الجانبي
  والبنفسجي الفاتح `#A5ADD1` للتأكيد، محتوى فاتح هادئ

## التشغيل

```bash
cp .env.example .env.local   # عبّئ قيم Supabase
npm install
npm run dev                  # http://localhost:3000
```

| المتغير | المصدر |
|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | Project Settings → API |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Project Settings → API |
| `SUPABASE_SERVICE_ROLE_KEY` | Project Settings → API — **للخادم فقط** |

> مفتاح Service Role لا يصل للمتصفح إطلاقاً: يُستخدم داخل Server Actions
> محمية بـ `requireSuperAdmin()` فقط (ملف `src/lib/supabase/admin.ts` معلّم
> بـ `server-only`).

## النشر على Vercel

1. اربط المستودع بمشروع Vercel جديد.
2. أضف متغيرات البيئة الثلاثة في Project Settings → Environment Variables.
3. انشر — لا إعدادات إضافية مطلوبة.

## ملاحظات أمان مطبقة

- الحماية ثلاثية الطبقات: Middleware (الجلسة) → `requireSuperAdmin` في كل
  صفحة وAction إداري → RLS في قاعدة البيانات (المرحلة 0)
- تغيير القسم أو الحالة أو الأدوار غير ممكن من جانب العميل — سياسات RLS تمنعه
  حتى لو تجاوز أحدهم الواجهة
- إنشاء المستخدم Atomic: فشل إنشاء Profile يحذف حساب Auth تلقائياً

## ما أضافته المرحلة 2

- **بنك المعلومات** `/knowledge`: قائمة المعلومات المنشورة المرئية للمستخدم فقط
  (RLS يفلتر تلقائياً)، مع فلترة حسب النوع وشارات «جديد | محدث | عاجل» خلال
  مدة الظهور التي حددها المضيف
- **البحث** المدمج في نفس الصفحة عبر دالة `search_knowledge` — عربي مطبّع
  (البحث بـ«اجراء» يجد «إجراء») ويحترم الصلاحيات والاستهداف
- **معلوماتي** `/contribute`: قائمة مساهمات المستخدم بحالاتها التسع، مع ملاحظة
  المدير كاملة عند الإعادة للتعديل
- **نموذج المعلومة** `/contribute/new`: البيانات الأساسية، المحتوى ثنائي اللغة،
  الوسوم، الأهمية والسرية (7 مستويات)، الجمهور المستهدف الديناميكي
  (قطاعات/أقسام/أدوار/مستخدمين حسب الاختيار)، وإعدادات النشر كاملة
  (مدة الظهور 1–10، انتهاء الصلاحية، تأكيد القراءة، الإشعارات، التنزيل/الطباعة)
- **المرفقات**: رفع مباشر إلى Bucket الخاص من صفحة تعديل المسودة، والحذف
  ينظف الملف من التخزين أيضاً
- **الإرسال للاعتماد** عبر دالة قاعدة البيانات `submit_for_approval` — يشعر
  مدير القسم تلقائياً
- **صفحة التفاصيل** `/knowledge/[id]`: المحتوى، البطاقة التعريفية (الرقم المرجعي،
  الإصدار، تواريخ النشر والسريان، القسم)، سبب التحديث وملخص التغييرات، المرفقات
  عبر روابط موقعة مؤقتة (تُنشأ فقط إن سمحت `can_view_version`)، تسجيل المشاهدة
  التلقائي، **وزر تأكيد القراءة الصريح** المنفصل عن مجرد الفتح
- **سجل الإصدارات** في صفحة التفاصيل، مع فتح أي نسخة سابقة بتنبيه واضح أنها
  مستبدلة ورابط للنسخة المعتمدة الحالية

## ما أضافته المرحلة 3

- **طلبات الاعتماد** `/approvals` (لمدير القسم ومدير النظام): الطلبات المعلقة
  بترتيب الأقدمية مع موعد SLA (يتحول أحمر عند التجاوز) وعداد إعادة التقديم،
  وسجل آخر 20 قراراً
- **صفحة المراجعة** `/approvals/[id]`: المحتوى الكامل، البطاقة الوصفية،
  ملخص الاستهداف، المرفقات، ورابط مقارنة مع الإصدار المنشور الحالي عند التحديث
- **شريط القرار**: اعتماد ونشر (مع تعديلات المدير الاختيارية على مدة العرض
  والأهمية قبل النشر) / إعادة للتعديل / رفض — الملاحظة إلزامية للإعادة والرفض
  في الواجهة وقاعدة البيانات معاً، وزر الاعتماد معطل تلقائياً لمعلومات المدير
  نفسها (والقاعدة ترفضها أيضاً)
- **إنشاء إصدار جديد** من صفحة أي معلومة منشورة (لمضيفي القسم ومديره): نوع
  التحديث (بسيط x.y+1 / جوهري x+1.0) وسبب التحديث وملخص التغييرات إلزامية،
  ثم فتح المسودة الجديدة بمحتوى واستهداف منسوخين جاهزين للتعديل
- **مقارنة الإصدارات** `/knowledge/[id]/compare`: فروقات على مستوى الكلمات
  (إضافات خضراء وحذوفات حمراء) للعنوان والملخص والمحتوى، وجدول تغييرات
  البيانات الوصفية — مع روابط مقارنة جاهزة من سجل الإصدارات
- **الأرشفة**: مدير القسم/النظام يؤرشف المعلومة بحوار تأكيد فتختفي من بنك
  المعلومات مع بقاء سجلها كاملاً، مع إمكانية إلغاء الأرشفة

## ما أضافته المرحلة 5 — تكامل WhatsApp

> **مطلوب**: نفّذ Migration `supabase/migrations/0006_whatsapp_worker.sql`
> (دالة الحجز الذري للطابور — مختبرة محلياً: لا ازدواج مع تعدد العمال،
> وممنوعة على عملاء الواجهة).

### المكونات

- **`supabase/functions/whatsapp-worker`**: معالج الطابور — حجز ذري
  (SKIP LOCKED)، تصيير القالب بمتغيراته، الإرسال عبر المزود، إعادة محاولة
  تصاعدية (5 دقائق × مربع المحاولة) حتى `max_attempts` ثم `failed`، تسجيل كل
  انتقال في `whatsapp_delivery_logs`، وتنبيه عاجل لمديري النظام عند تجاوز
  نسبة الفشل المحددة في الإعدادات (بدون تكرار خلال 24 ساعة)
- **`supabase/functions/whatsapp-webhook`**: استقبال حالات التسليم من Meta
  (sent/delivered/read/failed) مع التحقق الأولي بـ `hub.verify_token`
  وعدم التراجع لو وصلت الحالات بترتيب معكوس
- **`supabase/functions/_shared/providers.ts`**: واجهة المزود —
  `MockProvider` (تطوير، لا إرسال فعلي) و`CloudApiProvider`
  (Meta الرسمي حصراً). الاستبدال من لوحة التحكم دون أي تغيير في الكود،
  وأي نقص في متغيرات البيئة يعيد تلقائياً للمزود الوهمي بأمان
- **لوحة الإدارة** `/admin/whatsapp`: تفعيل/تعطيل التكامل واختيار المزود،
  إحصاءات الحالات الخمس، تحرير القوالب الست (عربي/إنجليزي × ثلاثة أحداث)
  مع تلميح المتغيرات المتاحة، وسجل آخر 50 رسالة بأرقام مقنّعة وسبب الفشل

### النشر والتشغيل

```bash
# 1) الأسرار (لا تضعها في الكود إطلاقاً)
supabase secrets set CRON_SECRET=<قيمة-عشوائية-قوية>
supabase secrets set WHATSAPP_PHONE_NUMBER_ID=...   WHATSAPP_BUSINESS_ACCOUNT_ID=...   WHATSAPP_ACCESS_TOKEN=...   WHATSAPP_API_VERSION=v20.0   WHATSAPP_WEBHOOK_VERIFY_TOKEN=<قيمة-تختارها>

# 2) نشر الدوال
supabase functions deploy whatsapp-worker
supabase functions deploy whatsapp-webhook --no-verify-jwt

# 3) الجدولة (Dashboard → Edge Functions → Schedules) كل دقيقة:
#    POST /functions/v1/whatsapp-worker مع ترويسة x-cron-secret: <CRON_SECRET>

# 4) في Meta for Developers → WhatsApp → Configuration:
#    Callback URL: https://<PROJECT>.functions.supabase.co/whatsapp-webhook
#    Verify token: نفس WHATSAPP_WEBHOOK_VERIFY_TOKEN، واشترك في حدث messages
```

ثم من لوحة `/admin/whatsapp`: فعّل التكامل واختر المزود الرسمي.
النظام يعمل قبل ذلك كاملاً بالمزود الوهمي — الرسائل تدخل الطابور وتُعالج
وتُسجل دون إرسال فعلي.

> ملاحظة إنتاجية: عند إضافة `APP_SECRET` من Meta يُوصى بتفعيل التحقق
> بتوقيع `X-Hub-Signature-256` في الـWebhook (مُعلَّم في الكود).
